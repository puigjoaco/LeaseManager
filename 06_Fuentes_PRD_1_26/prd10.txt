# Product Requirements Document: Rent Control
*Última actualización: 29 de Mayo de 2025 (incorporando todas las clarificaciones definitivas post-análisis exhaustivo, con decisiones finales inamovibles sobre todos los aspectos del sistema)*

# Overview
Rent Control es una aplicación web de gestión integral, diseñada a medida para automatizar y centralizar completamente la administración de arriendos de locales comerciales en Chile. Está dirigida principalmente al `Administrador` del negocio, proporcionándole herramientas para eliminar procesos manuales, reducir errores y obtener un control financiero y operativo completo. También ofrece acceso de solo lectura a los `Socios` del negocio (máximo ~15 usuarios) y a la `Contadora` (según se defina en las fases de desarrollo). El sistema resuelve los problemas asociados a la gestión manual actual, como cálculos de renta complejos y propensos a errores, envío manual de notificaciones, conciliación bancaria tediosa, información descentralizada y seguimiento manual de plazos y pagos de servicios. Su valor principal es la automatización, la centralización de datos (incluyendo documentos), la reducción de errores y la mejora significativa en la eficiencia y control del negocio de arriendos. El sistema está diseñado para manejar aproximadamente 17 usuarios totales (15 socios + 1 contadora + 1 administrador).

# Core Features

* **Gestión Centralizada de Datos Fundamentales:**
   * *Qué hace:* Permite administrar (crear, ver, editar, eliminar) de forma centralizada toda la información clave:
       * `Socios`: Información personal y de contacto de los socios del negocio (ej. Nombre, RUT, Email, Teléfono, Dirección). Los socios no pueden ser eliminados si tienen participaciones activas; en su lugar, se ofrece un wizard de transferencia para mantener siempre el 100% de participación. Un socio puede tener 100% de participación en múltiples empresas o propiedades diferentes sin restricción global.
       * `Empresas`: Datos de las sociedades de inversión, incluyendo razón social, RUT, dirección. Se gestiona su(s) cuenta(s) bancaria(s) asociada(s) y la composición de socios con sus respectivos porcentajes de participación (que deben sumar exactamente 100%). Al crear una nueva empresa, se le debe asociar al menos una cuenta bancaria activa. El sistema NO permitirá guardar una empresa si la suma de porcentajes no es exactamente 100%. Si una empresa pierde todas sus cuentas bancarias activas, debe asignársele una nueva antes de poder operar. Si la empresa no tiene propiedades, contratos ni cuentas bancarias activas, se permite eliminación directa sin proceso de disolución.
       * `Propiedades`: Detalles completos de cada inmueble, incluyendo un código único de 3 dígitos (numérico, ej: 001, 045, 234), RUT de avalúo fiscal, dirección, ciudad, datos de servicios (agua, luz, gas con nombre de empresa y número de cliente), información de gastos comunes (si tiene y detalles relevantes). Una propiedad debe poder asignarse como perteneciente a una `Empresa` específica *o* a un conjunto de `Socios` individuales (donde la suma de sus porcentajes de propiedad debe ser exactamente 100%, siendo estos socios los definidos en la entidad general `Socios`). No puede pertenecer a ambos simultáneamente. No pueden existir dos propiedades con el mismo código. Cada propiedad tiene UN solo contrato vigente a la vez, aunque puede tener UN ÚNICO contrato futuro programado (máximo). Cuando existe conflicto entre renovación automática y contrato futuro, el Administrador debe decidir manualmente cuál mantener.
       * `Arrendatarios`: Información de quienes arriendan las propiedades. Pueden ser personas naturales (requiriendo Nombre, Apellido, RUT Persona, Email, Teléfono, Dirección de notificaciones) o empresas (requiriendo Razón Social, RUT Empresa, Nombre Representante Legal, RUT Representante Legal, Email, Teléfono, Dirección de notificaciones). El sistema debe permitir seleccionar el tipo y mostrar los campos correspondientes. El campo Dirección es obligatorio para poder enviar cartas certificadas de término. El teléfono debe ingresarse en formato internacional obligatorio: +(código país)(número) sin espacios.
       * `Cuentas Bancarias`: Datos de las cuentas bancarias utilizadas, como Nombre del Banco, Número de Cuenta, Nombre del Titular, RUT del Titular, Email de Contacto. Una cuenta bancaria puede estar asociada a múltiples `Empresas` que la utilizan para mantener sus dineros.
       * `Credenciales Bancarias`: Almacenamiento seguro de las credenciales de acceso a los portales bancarios (Nombre de Usuario, Contraseña), asociadas a una `Cuenta Bancaria` específica para la funcionalidad de conciliación. Solo el Administrador puede ver o modificar estas credenciales. Incluye sistema de recuperación mediante preguntas de seguridad (3 preguntas configuradas al crear las credenciales, requiere responder correctamente 2 de 3 para recuperar acceso). Las 3 preguntas son SOLO para credenciales bancarias. El acceso al sistema se recupera por email. Incluye la gestión del tipo de Autenticación de Dos Factores (2FA) y la información asociada:
           * `tipo_2fa_configurado`: Opciones como 'NO_2FA' (Sin 2FA - opción por defecto o muy clara), 'TOTP' (para códigos de apps autenticadoras), 'SMS_ASISTIDO', 'PUSH_ASISTIDO', 'OTRO_ASISTIDO'. El `Administrador` seleccionará la opción correspondiente.
           * `secreto_totp_encriptado`: Campo para almacenar el secreto TOTP (si `tipo_2fa_configurado` es 'TOTP'), encriptado con AES-256-GCM. Este campo solo será visible/editable en la UI si se selecciona 'TOTP'.
           * `notas_2fa_asistida`: Campo para notas del administrador (ej. a quién contactar para un código SMS) si `tipo_2fa_configurado` es de tipo asistido. Este campo solo será visible/editable en la UI si se selecciona un tipo de 2FA asistido.
           * El sistema de scraping toma las credenciales encriptadas y ejecuta todo el proceso automáticamente, incluyendo manejo de 2FA según el tipo configurado, sin intervención humana.
           * El Administrador podrá actualizar estas credenciales de forma segura a través de una interfaz dedicada.
       * `Contratos`: Todos los términos contractuales incluyendo estado (ej. 'Vigente', 'Pendiente Activación', 'Finalizado', 'Cancelado', 'Terminado Anticipadamente'), la gestión detallada del ciclo de vida de la `garantía`, `dia_pago_mensual` (entre día 1 y 5, nunca día 31), `plazo_notificacion_termino_dias` (campo obligatorio que indica con cuántos días de anticipación al término del contrato se debe notificar la no renovación), tipo de notificación (Solo Email, Email + WhatsApp), y la estructura detallada de períodos contractuales, reajustes y ajustes específicos. Incluye funcionalidad para "Terminación Anticipada de Contrato" y "Gestión de Avisos de Término" con posibilidad de cancelación. Los contratos solo pueden iniciar el día 1 de cada mes y siempre terminan el último día del mes. Incluye gestión documental para almacenar el PDF del contrato firmado con firma electrónica. **Sistema de Períodos**: Un contrato puede tener múltiples períodos (inicial + renovaciones), manteniendo todo el historial en un solo contrato en lugar de crear contratos nuevos. **Contratos con Tramos**: Opción para habilitar arriendos con valores diferentes por períodos (ej: meses 1-3: $900,000, meses 4-7: 30 UF, meses 8+: 32 UF). **Validación de Monto Mínimo ABSOLUTO**: NO EXISTE ningún contrato con monto mensual menor a $1,000 CLP. El sistema RECHAZA automáticamente cualquier intento de crear contratos con montos inferiores. Esta regla es inamovible y definitiva. Para contratos en UF, la validación es al crear: No se puede ingresar un valor en UF que HOY sea < $1,000. **Contratos Retroactivos**: Se permite crear contratos con CUALQUIER fecha de inicio en el pasado, sin límite temporal. El sistema solo creará pagos mensuales desde la fecha de creación hacia adelante. **Cambio de Arrendatario con Contrato Futuro**: El cambio de titular se permite y el contrato futuro se mantiene intacto. NO se modifica, NO se cancela, NO se notifica sobre el contrato futuro.
       * `PlantillaContrato`: (Nueva entidad) Permite al `Administrador` crear y gestionar plantillas reusables de contratos con parámetros predefinidos comunes. Campos: `nombre_plantilla`, `valor_base_periodo_inicial`, `tipo_moneda_base_inicial`, `porcentaje_reajuste_maestro` (almacenado como porcentaje directo, ej: 3.00 para 3%), `periodicidad_reajuste`, `dia_pago_mensual`, `plazo_notificacion_termino_dias`, `garantia_monto_pactado`, `duracion_meses_default`.
       * `CategoriaMovimiento`: (Nueva entidad) Permite al `Administrador` gestionar las categorías para `IngresosManuales` y `GastosManuales`. Campos: `nombre_categoria` (único), `tipo_movimiento` ('Ingreso' o 'Gasto'). El sistema se entregará con categorías predefinidas como "Recepción de Garantía del Contrato", "Devolución de Garantía del Contrato", "Contribuciones", "Reparaciones", "Devolución pago excedente", etc., y el Administrador podrá añadir/editar/eliminar categorías a través de una interfaz de gestión.
   * *Por qué es importante:* Establece la única fuente de verdad para todos los datos del negocio, permitiendo la integridad y consistencia de la información.
   * *Cómo funciona a alto nivel:* Interfaces web para ingresar/modificar datos con validaciones de integridad (RUTs válidos, suma de porcentajes de socios en Empresas y Propiedades = exactamente 100%, campos obligatorios según tipo de Arrendatario, validación ABSOLUTA de monto mínimo de $1,000 para contratos, formato internacional de teléfonos, etc.). Interfaz segura para que el Administrador actualice credenciales bancarias con sistema de recuperación. Interfaz de administración para `CategoriaMovimiento`. Sistema de plantillas para agilizar la creación de contratos. Proceso de disolución de empresa requiere eliminar primero todos los contratos activos, luego las propiedades, y finalmente la empresa, permitiendo crear nuevas propiedades asignadas a socios como personas naturales (excepto si la empresa no tiene activos, en cuyo caso se permite eliminación directa). Gestión de conflictos de renovación permite al Administrador elegir entre mantener la renovación automática o el contrato futuro programado. Sistema de transferencia o cambio de arrendatario permite crear nuevo contrato o mantener el existente con nuevo titular según decisión del Administrador.

* **Gestión Avanzada de Períodos Contractuales y Ajustes de Renta:**
   * *Qué hace:* Permite la definición inicial de los términos contractuales que rigen el cálculo del arriendo a lo largo de su ciclo de vida y la aplicación flexible de ajustes (descuentos o recargos) sobre el `Monto Contractual Bruto del Período` calculado. Un contrato se estructura como una secuencia de `PeriodoContractual`, donde cada uno representa un tramo de tiempo con un monto de arriendo bruto específico. **Sistema de Períodos de Renovación**: Cuando un contrato se renueva automáticamente, se agregan nuevos períodos al mismo contrato en lugar de crear uno nuevo.
   * **Definición de Parámetros Fundamentales del Contrato (Modificables con Versionado):**
       * *Parámetros Definibles al Crear/Configurar el Contrato:*
           * `fecha_inicio_contrato` y `fecha_fin_contrato`: Definen la duración total del término contractual vigente (inicial o una renovación). Solo se permiten fechas de inicio el día 1 de cada mes. Las fechas de fin siempre son el último día del mes.
           * `valor_base_periodo_inicial` (en UF o CLP): Es el monto de arriendo bruto para el primer tramo (el primer `PeriodoContractual`) del contrato. Este valor se ingresa directamente y NUNCA puede ser menor a $1,000 CLP.
           * `porcentaje_reajuste_maestro` (decimal, puede ser 0 si no hay reajustes): Es el porcentaje que se aplicará para calcular los montos de los `PeriodoContractual` subsecuentes. El usuario ingresa el valor numérico del porcentaje deseado (ej. `3` para un 3%); el sistema lo almacena directamente como 3.00 y para los cálculos lo divide por 100.
           * `periodicidad_reajuste` (ej. 'Anual', 'Semestral', 'Trimestral'): Define la frecuencia con la que se aplica el `porcentaje_reajuste_maestro`.
           * `tiene_tramos` (boolean): Indica si el contrato utiliza el sistema de tramos con valores diferenciados por período.
       * **Versionado y Modificaciones:** Los parámetros contractuales pueden ser modificados mediante un proceso de aprobación especial. Cada modificación genera una nueva versión del contrato con trazabilidad completa en el `AuditLog`. Los `PagoMensual` ya generados mantienen los valores con los que fueron calculados originalmente. La `fecha_fin_contrato` puede ser modificada a través de la funcionalidad de "Terminación Anticipada de Contrato".
       * **Contratos con fecha retroactiva:** Se permite crear contratos con CUALQUIER fecha de inicio en el pasado sin límite temporal. El sistema actualiza los períodos y valores según corresponda pero NO crea pagos mensuales retroactivos. Los pagos mensuales solo se crean desde la fecha de creación del contrato hacia adelante. El sistema ignora completamente los meses pasados y empieza a crear historial desde la creación del contrato.
   * **Sistema de Tramos de Arriendo (Opcional):**
       * Permite definir valores de arriendo diferentes para distintos períodos del contrato
       * Cada tramo especifica: mes inicio, mes fin (opcional), tipo de valor (UF/CLP), y monto
       * Ejemplo: Meses 1-3: $900,000 CLP, Meses 4-7: 30 UF, Meses 8+: 32 UF
       * Los tramos se pueden habilitar/modificar después de crear el contrato
       * Al renovar un contrato con tramos, se usa el valor del último tramo como base, sin mantener la estructura de tramos
   * **Renovación Automática del Contrato:**
       * Si transcurre el `plazo_notificacion_termino_dias` sin que ninguna de las partes registre un aviso de término, el contrato se renueva automáticamente a las 02:01 del día 1 del mes siguiente al vencimiento.
       * Se crea un nuevo conjunto de `PeriodoContractual` para el período renovado, aplicando el `porcentaje_reajuste_maestro` sobre el último valor del período anterior (o último tramo si tiene tramos).
       * El contrato mantiene su estado 'Vigente' y se marca el tipo de período como 'Renovación Automática'.
       * Se notifica al Administrador según su configuración personal: canal (email, WhatsApp, ambos o ninguno) y hora del día (configurable dentro del software).
       * Si existe un contrato futuro programado para la misma propiedad, se genera un conflicto que requiere decisión manual del Administrador entre renovar el contrato actual (cancelando el futuro) o mantener el contrato futuro programado.
   * **Generación y Cálculo de `PeriodoContractual` y su `Monto Contractual Bruto`:**
       * El sistema genera una secuencia de registros de `PeriodoContractual` para cubrir la duración total del contrato (desde `fecha_inicio_contrato` hasta `fecha_fin_contrato`), segmentando el tiempo según la `periodicidad_reajuste`.
       * Para el primer `PeriodoContractual` (el tramo inicial del contrato), su `Monto Contractual Bruto del Período` es el `valor_base_periodo_inicial` definido en el `Contrato`.
       * Para cada `PeriodoContractual` subsiguiente, su `Monto Contractual Bruto del Período` se calcula aplicando el `porcentaje_reajuste_maestro` al `Monto Contractual Bruto del Período` del `PeriodoContractual` inmediatamente anterior.
       * En caso de renovación automática, el primer período de la renovación toma como base el último monto del período anterior y aplica el reajuste correspondiente.
   * **Visualización y Planificación:** La interfaz permitirá definir los parámetros del contrato (`valor_base_periodo_inicial`, `porcentaje_reajuste_maestro`, `periodicidad_reajuste`, fechas del término) y visualizará la secuencia proyectada de `PeriodoContractual` con sus fechas y los `Montos Contractuales Brutos` calculados. Mostrará claramente los períodos renovados automáticamente con su tipo y origen.
   * **Gestión Simplificada de Ajustes sobre el `Monto Contractual Bruto del Período` (Descuentos/Recargos):**
       * Una vez calculado el `Monto Contractual Bruto del Período` para un `PeriodoContractual` específico, toda variación sobre este monto para determinar el `Monto Neto a Cobrar` se gestionará a través de un único tipo de `AjusteContrato`:
       * **Ajuste Unificado:**
           * Solo el Administrador puede crear, modificar o eliminar ajustes
           * Permite aplicar una variación (descuento o recargo, en UF o CLP) al `Monto Contractual Bruto del Período`.
           * *Parámetros:* Tipo (Descuento/Recargo), Monto del Ajuste, Tipo de Monto (UF/CLP), Mes Inicio, Mes Fin (opcional), Justificación (obligatoria).
           * Los ajustes se aplican por mes completo, NO por días. Un ajuste de $100,000 es el mismo monto sin importar si el mes tiene 28, 30 o 31 días.
           * Si `mes_fin` es NULL, el ajuste es indefinido (continúa hasta ser cancelado).
           * Los ajustes se aplican ANTES de agregar el código de propiedad para no afectar la identificación del pago.
           * **REGLA ABSOLUTA DE AJUSTES**: Un contrato de $1,000 NO PUEDE recibir ningún ajuste de descuento. Para contratos > $1,000, el ajuste máximo de descuento permitido es: (monto_actual - 1,000). Ejemplo: Contrato de $2,000 → descuento máximo $1,000 → resultado mínimo $1,000. El sistema RECHAZA automáticamente cualquier ajuste que resulte en un monto neto menor a $1,000 CLP.
           * **Ajustes Multi-Moneda**: Se permiten ajustes en cualquier moneda sobre cualquier contrato. Si el ajuste está en una moneda diferente al contrato, la conversión usa el valor UF del día 1 del mes correspondiente al ajuste.
           * Validación de fechas: mes_fin debe ser mayor o igual a mes_inicio si está definido.
       * Al guardar cualquier ajuste, este y su justificación se propagan inmediatamente a los módulos relevantes (`PagoMensual`, mensajería, reportes) para los meses afectados.
       * **Recálculo en Tiempo Real:** Si se aplica un ajuste durante el día, el sistema recalcula inmediatamente el `PagoMensual` afectado. Si ya se enviaron notificaciones, los cambios aplican desde la próxima comunicación.
   * *Por qué es importante:* Asegura la trazabilidad y versionado de los términos contractuales, gestiona automáticamente las renovaciones, y ofrece máxima flexibilidad para manejar situaciones particulares a través de un sistema de ajustes simplificado.
   * *Cómo funciona a alto nivel:*
       * La interfaz de configuración de contratos permitirá definir los parámetros base o seleccionar una plantilla existente.
       * El sistema mostrará todos los períodos del contrato (inicial + renovaciones) en una vista unificada.
       * Proceso automático verifica diariamente a las 02:01 AM del día 1 contratos que terminaron el último día del mes anterior y ejecuta renovaciones cuando corresponde.

* **Gestión de Avisos de Término:**
   * *Qué hace:* Permite registrar y gestionar los avisos de término de contrato, tanto enviados por el Administrador como recibidos del arrendatario, con posibilidad de cancelación.
   * **Funcionalidades:**
       * **Registrar Aviso de Término Enviado:**
           * Genera carta de término automática en PDF con los datos del contrato
           * Registra fecha de envío
           * Envía por email/WhatsApp al arrendatario
           * Adjunta comprobante de envío
           * Marca el contrato para NO renovación automática
       * **Registrar Aviso de Término Recibido:**
           * Registra fecha de recepción
           * Permite subir carta escaneada
           * Registra medio de recepción (email, carta física, etc.)
           * Marca el contrato para NO renovación automática
           * Se permiten fechas pasadas pero se marca como "Registro Tardío" en auditoría
       * **Cancelar Aviso de Término:**
           * Permite cancelar un aviso previamente registrado
           * Requiere motivo de cancelación
           * Reactiva la renovación automática
           * Registra auditoría completa
   * **Validaciones:**
       * **Validación Temporal Definitiva**: El aviso es válido si el timestamp del registro es <= 23:59:59 del último día del plazo (`plazo_notificacion_termino_dias`). El sistema registra con precisión de segundos.
       * Alerta si el aviso está fuera de plazo
       * Impide renovación automática si existe aviso registrado activo
   * *Por qué es importante:* Formaliza el proceso de término de contratos y previene renovaciones automáticas no deseadas, con flexibilidad para cambios de decisión.
   * *Cómo funciona:* Interfaz específica en la ficha del contrato con opciones claras para gestionar avisos, integrada con el sistema de renovación automática.

* **Cálculo Automático y Preciso de Renta Mensual:**
   * *Qué hace:* Cada día 1 del mes a las 02:01 AM (para evitar problemas con cambio de horario), calcula el monto exacto en CLP a cobrar por cada contrato activo DEL mes actual (para cobrar en el mes) y cierra definitivamente el mes anterior. Para volúmenes menores a 500 propiedades, utiliza procesamiento síncrono simple. Los cálculos incluyen validación de monto mínimo para asegurar al menos 4 dígitos (>= $1,000 CLP) para el código de propiedad. El valor de la UF se obtiene con aproximadamente 20 días de anticipación (día 10 del mes anterior) y se almacena en la base de datos, garantizando que siempre esté disponible para el cálculo. El proceso implica los siguientes pasos secuenciales:
       1.  Identificar el `PeriodoContractual` vigente para el mes en curso y obtener su `Monto Contractual Bruto del Período` (en su moneda original, UF o CLP).
       2.  **Obtención del `Monto Neto de Arriendo del Mes` en CLP:**
           a.  Se toma el `Monto Contractual Bruto del Período` (del paso 1).
           b.  Si este monto bruto está en UF, se convierte a CLP utilizando el valor de la UF correspondiente al día 1 del mes en curso (obtenido previamente el día 10 del mes anterior). **Regla del Valor UF Más Alto**: Para contratos retroactivos, se analiza TODO el período desde la fecha de inicio hasta hoy y se usa el valor UF más alto histórico encontrado. Si el valor UF actual es menor que el máximo histórico del contrato, se usa el valor más alto histórico. Si ya está en CLP, se utiliza ese valor.
           c.  Al monto en CLP resultante se le aplica cualquier `AjusteContrato` que sea aplicable al mes en curso. Los ajustes se aplican por mes completo y ANTES del código de propiedad.
           d.  El resultado de esta operación es el `Monto Neto de Arriendo del Mes` en CLP.
       3.  Obtención del valor de la UF: El sistema obtiene este valor aproximadamente 20 días antes del día 1 (el día 10 del mes anterior) de una lista priorizada de APIs externas configuradas. Si todas las fuentes automáticas fallaran, se habilitará la entrada manual para el Administrador.
       4.  Del monto neto en CLP, se toma únicamente la parte entera, descartando (truncando) todos los decimales.
       5.  **REGLA DEFINITIVA E INAMOVIBLE DEL CÓDIGO DE PROPIEDAD**: Al monto entero resultante, se le aplica el ajuste especial que reemplaza sus últimos 3 dígitos enteros por el `Código de Propiedad` único de 3 dígitos asociado al inmueble. El sistema SIEMPRE acepta cualquier pérdida monetaria resultante de esta operación, sin importar el monto. Si un arriendo es de $45,000,000 y el código es 045, se cobra $45,000,045 y se pierde $44,999. Esta es una decisión de negocio final e incuestionable. El sistema siempre pierde o mantiene neutralidad, nunca genera ganancia por este concepto. Para montos menores a $1,000,000, se aplica concatenación directa: $850,000 + código 123 = $850,123. Para montos de cualquier tamaño, SIEMPRE se reemplazan los últimos 3 dígitos sin importar el tamaño del monto.
       6.  El resultado de esta sustitución es el `monto_calculado_clp` final que se guarda en el registro `PagoMensual`.
   * **Procesamiento Simple para <500 Propiedades:**
       * Ejecución síncrona directa sin Celery
       * Un loop simple que procesa todos los contratos
       * Sistema de alertas al llegar a 450 propiedades
       * Activación de Celery en paralelo al llegar a 480 propiedades
       * Migración automática a procesamiento asíncrono programada para el próximo día 1 al superar 500 propiedades
       * Si falla Redis, el cálculo procede sin caché en modo degradado
   * **Cierre de Mes:**
       * **Cierre Definitivo**: El cierre automático ocurre el último día del mes a las 23:59:59
       * Los movimientos bancarios posteriores no afectarán el mes cerrado automáticamente
       * Solo el Administrador puede reabrir un mes cerrado con justificación obligatoria
       * Toda modificación en un mes cerrado genera entrada detallada en AuditLog
       * Permite asignación retroactiva de pagos a meses cerrados cuando el scraping no detectó el pago originalmente
       * **Asignación de Mes por Timestamp**: El timestamp de INICIO de la transferencia bancaria determina el mes contable. Si fue iniciada el 31/05 a las 23:58, cuenta para mayo sin importar cuándo el banco la procesó.
   * *Por qué es importante:* Automatiza el cálculo crítico de la renta mensual con un identificador único para conciliación bancaria perfecta.
   * *Cómo funciona a alto nivel:* Proceso automático ejecutado por cron job que calcula secuencialmente todos los arriendos del mes, con sistema de pre-carga del valor UF.

* **Generación y Seguimiento de Cobros Mensuales (`PagoMensual`):**
   * *Qué hace:* Crea/almacena un registro (`PagoMensual`) por mes de arriendo activo. Este registro contendrá campos como: `contrato_id`, `periodo_contractual_id` (FK al `PeriodoContractual` vigente para ese mes), `mes_correspondiente`, `anio_correspondiente`, `monto_contractual_bruto_original_periodo`, `tipo_moneda_bruto_original_periodo`, `monto_neto_arriendo_mes_original`, `tipo_moneda_neto_original`, `valor_uf_aplicado`, `valor_uf_historico_maximo`, `monto_calculado_clp`, `fecha_vencimiento`, `fecha_pago`, `estado_pago`, `ids_ajustes_aplicados`, `justificacion_ajustes_aplicados`, y `mes_cerrado`. Su estado se actualiza automáticamente mediante el proceso de conciliación bancaria o manualmente como fallback.
   * **Sistema de No Pagos Parciales en Conciliación Automática:** El sistema solo acepta pagos completos del monto mensual total en el proceso automático. La conciliación automática NUNCA une automáticamente múltiples transferencias. Solo acepta matches exactos entre transferencia única y monto esperado. Todas las transferencias sin match exacto se registran como IngresoDesconocido. El Administrador debe manualmente seleccionar múltiples IngresoDesconocido, verificar que suman el monto exacto y asignarlos al PagoMensual correspondiente.
   * **Sistema de Reversión de Pagos:** Incluye funcionalidad para revertir pagos marcados erróneamente como PAGADO, actualizando en cascada todos los registros relacionados.
   * `fecha_vencimiento`: Se establece entre el día 1 y 5 del mes según el `dia_pago_mensual` del `Contrato` asociado. **Regla de Días de Atraso Definitiva**: El sistema ejecuta conciliación SIEMPRE el día programado (1-5). NO existe concepto de "feriado bancario" para el sistema. Si día 5 = sábado/domingo/feriado → conciliación ejecuta igual. Si no hay dinero en la cuenta el día 5 → ATRASADO. El timestamp de INICIO de transferencia determina el mes contable. Responsabilidad del arrendatario: asegurar que el dinero esté disponible.
   * *Por qué es importante:* Control exacto de cobros con política estricta de pagos completos en conciliación automática, pero con flexibilidad manual para casos especiales.
   * *Cómo funciona a alto nivel:* Generado por el proceso de cálculo mensual. Actualizado por conciliación automática o entrada manual del Administrador.

* **Sistema de Gestión de Deudas y Repactación:**
   * *Qué hace:* Mantiene un registro claro y detallado de todas las deudas de los arrendatarios y permite repactar deudas en cuotas.
   * **Registro de Deudores:**
       * Vista consolidada por arrendatario mostrando:
           * Meses adeudados (lista detallada)
           * Monto exacto por cada mes
           * Deuda total acumulada
           * Días de atraso por cada mes
           * **Score de Pago Definitivo**: Calculado como promedio simple (pagos a tiempo / total pagos CON DATOS). El sistema IGNORA completamente los meses sin registro. Considera pago "a tiempo" hasta el día de vencimiento inclusive. Para contratos retroactivos, el sistema marca claramente "Score calculado desde: [primer mes con datos]" y calcula el score solo con los datos existentes en el sistema.
           * Detalle histórico de días de pago de cada mes
       * Historial completo de deudas y pagos
   * **Repactación de Deudas:**
       * **Regla Definitiva**: La repactación SIEMPRE es por el total de la deuda acumulada. No se permiten repactaciones parciales.
       * Permite dividir la deuda total en cuotas configurables
       * Las cuotas se suman automáticamente a los arriendos futuros como un recargo
       * **Aplicación del Código con Repactación**: Ejemplo detallado: Si el arriendo base es $1,200,920 y la cuota de repactación es $300,120, el valor total sería $1,501,040. Al aplicar el código de propiedad 045, se reemplazan los últimos tres dígitos, resultando en $1,501,045.
       * Registro completo del plan de pago:
           * Deuda original total
           * Número de cuotas
           * Valor por cuota
           * Calendario de pagos
       * Ejemplo: Deuda de $1,500,000 en 6 cuotas = $250,000 adicionales por mes sumados al arriendo base
   * *Por qué es importante:* Formaliza el manejo de atrasos y ofrece herramientas para ayudar a arrendatarios con dificultades de pago mientras mantiene el control.
   * *Cómo funciona:* Interfaz específica para gestionar deudas, integrada con el cálculo mensual para aplicar automáticamente las cuotas de repactación como recargo al arriendo.

* **Notificaciones Automáticas a Arrendatarios (Email y WhatsApp desde MVP):**
   * *Qué hace:* Envía comunicaciones vía Email y WhatsApp a los arrendatarios. Requiere templates pre-aprobados para WhatsApp Business. No procesa respuestas de arrendatarios (comunicación unilateral). **No permite opt-out**: las notificaciones son persistentes y obligatorias. Las notificaciones WhatsApp se envían siempre a las 10:00 AM en los días programados. El sistema de notificaciones se estructura en fases:
       * **Configuración por Contrato:**
           * Cada contrato define su tipo de notificación: "Solo Email" o "Email + WhatsApp"
           * El email SIEMPRE se envía
           * WhatsApp es opcional según configuración
       * **Fase de Cobro Inicial (Email y WhatsApp en MVP):**
           * **Día 1 del mes (10:00 AM):** Se envía un aviso inicial a todos los arrendatarios por los canales configurados con template profesional indicando monto exacto y plazo hasta el día 5 a las 13:59 hrs. Las notificaciones se envían sin importar si es feriado bancario, pero el sistema no penaliza atrasos en días feriados.
           * **Día 3 del mes (10:00 AM):** Primer recordatorio (solo pendientes).
           * **Día 5 del mes (10:00 AM):** Segundo recordatorio (solo pendientes).
       * **Fase de Seguimiento de Atrasos:**
           * **Días 10, 15, 20 y 25 (10:00 AM):** Notificaciones de atraso con mensaje específico.
       * **Manejo de Errores y Reintentos:**
           * Estados de entrega: Enviado → Entregado → Leído → Fallido
           * Reintentos inteligentes: 0min → 5min → 30min
           * **Notificaciones Fallidas Definitivo**: Después de 10 fallos consecutivos de WhatsApp se alerta al Administrador informando las fallas, pero el sistema continúa intentando indefinidamente. Email SIEMPRE se envía como respaldo, sin excepción. NUNCA se suspenden las notificaciones por email aunque WhatsApp falle.
           * Se registra en log todos los intentos y fallos
       * **Operaciones Masivas de Notificación:** 
           * Selección múltiple de arrendatarios
           * Envío masivo manteniendo personalización
   * **Log de Comunicaciones:** 
       * Registro completo de todas las comunicaciones enviadas
       * Tracking de estado de entrega
       * Historial por arrendatario consultable
       * Registro de fallos por canal (WhatsApp/Email) con contrato, propiedad y arrendatario afectado
   * **Formato de teléfonos:** Obligatorio formato internacional +(código país)(número) sin espacios. Sistema valida formato al crear/editar arrendatario.
   * *Por qué es importante:* Comunicación efectiva usando los canales preferidos en Chile, con trazabilidad completa y sin posibilidad de desuscripción.
   * *Cómo funciona a alto nivel:* Procesos programados que ejecutan el envío usando APIs de SendGrid/Mailgun y Twilio, con manejo robusto de errores y templates pre-aprobados.

* **Conciliación Bancaria Inteligente (con API como método principal):**
   * *Qué hace:* Conecta de forma segura a los bancos registrados utilizando **prioritariamente la API oficial del Banco de Chile**, con web scraping como método secundario de respaldo únicamente cuando la API falle o no esté disponible. La estructura del código intentará siempre la API primero, con fallback automático a scraping si falla. La API realizará las mismas funciones: obtener movimientos de ingresos y egresos para comparar con el sistema y mantener sincronizados los saldos. **Scraping Autorizado Definitivo**: El scraping está autorizado por el usuario con su banco y funciona automáticamente sin restricciones usando las credenciales encriptadas proporcionadas cuando la API no está disponible. El sistema toma las credenciales y ejecuta todo el proceso automáticamente, incluyendo manejo de 2FA, sin intervención humana.
   * **Proceso de Conciliación:**
       * Intenta conexión vía API Banco de Chile
       * Si API responde exitosamente → procesa movimientos
       * Si API falla (timeout, error 500, credenciales) → fallback automático a web scraping
       * Si ambos fallan → alerta al administrador para conciliación manual
       * Log detallado del método usado en cada conciliación
       * Intenta match automático por `monto_calculado_clp` único (sin tolerancia)
       * Identifica ingresos por garantías y otros movimientos
       * Compara saldos bancarios vs sistema
       * Alerta sobre movimientos no reconocidos
       * Para pagos incorrectos: se registra devolución completa del monto erróneo como gasto para que el arrendatario deposite el monto correcto
       * Múltiples pagos parciales se registran como ingresos desconocidos para asignación manual
   * **Manejo de Pagos Excedentes - PROCESO DEFINITIVO:**
       * Cuando un pago excede el monto esperado (ej: esperado $1,234,567, recibido $1,500,000):
       * Sistema crea automáticamente:
           * IngresoDesconocido por $1,500,000 con descripción "Pago excedente - Esperado: $1,234,567"
           * GastoManual por $1,500,000 con categoría "Devolución pago excedente" y descripción "Devolución total - pago incorrecto"
       * Los movimientos se netean a cero contablemente
       * Se notifica al arrendatario para que deposite el monto correcto ($1,234,567)
       * La conciliación posterior detecta el nuevo pago correcto y lo marca como pagado
   * **Conciliación Manual como Fallback:**
       * Interfaz para ingresar manualmente monto y fecha
       * Sistema intenta match automático con arriendos pendientes
       * Permite asignar múltiples ingresos desconocidos a una propiedad (para casos de transferencias múltiples que suman el total)
       * Si no hay match, permite categorizar como ingreso/gasto manual
       * Permite asignación retroactiva a meses cerrados (solo Administrador) con auditoría especial
   * **Timeline de Conciliación:**
       * Días 1-25: Conciliación automática diaria
       * Día 25: Última conciliación automática del mes
       * Días 26-30/31: Solo conciliación manual o por solicitud
       * Fin de mes: Conciliación final manual antes de transferir a dueños y dejar cuenta en cero
   * **Gestión de Credenciales Duales:**
       * API Credentials: API Key, Secret, Client ID (encriptados)
       * Scraping Credentials: Usuario, Password (encriptados)
       * Sistema detecta automáticamente qué credenciales usar según el método
       * Interfaz unificada que oculta la complejidad al administrador
   * **Estrategia de Fallback:**
       * Intento #1: API con timeout de 30 segundos
       * Si falla → Log del error específico
       * Intento #2: Web scraping automático
       * Si falla → Notificación al admin
       * Opción manual siempre disponible
   * **Monitoreo de Salud de API:**
       * Endpoint de health check cada 5 minutos
       * Dashboard muestra estado: "API ✓", "Scraping ✓", "Manual"
       * Métricas de éxito/fallo por método
       * Alertas si API falla consistentemente
   * *Por qué es importante:* Automatiza la conciliación priorizando métodos oficiales (API) pero manteniendo funcionalidad con scraping cuando sea necesario.
   * *Cómo funciona a alto nivel:* Proceso que intenta API primero, falla gracefully a scraping, y siempre permite intervención manual como último recurso.

* **Registro de Otros Ingresos y Gastos Manuales:**
   * *Qué hace:* Permite al Administrador registrar cualquier movimiento de dinero no relacionado directamente con el cobro automático de arriendos. Incluye gestión completa de garantías a través de categorías especiales y funcionalidad de operaciones masivas.
   * **Categorías del Sistema:**
       * "Recepción de Garantía del Contrato"
       * "Devolución de Garantía del Contrato"
       * "Contribuciones"
       * "Reparaciones"
       * "Devolución pago excedente"
       * Categorías personalizadas por el Administrador
   * **Operaciones Masivas:**
       * **Ingreso Masivo de Gastos Manuales**: Permite seleccionar múltiples gastos y asignar categoría común. Requisitos por cada gasto: Monto individual, Propiedad específica (obligatorio), Detalle descriptivo. El flujo consiste en seleccionar categoría (ej: "Contribuciones"), agregar N gastos con sus datos individuales, el sistema valida completitud antes de guardar y genera registro individual por cada gasto. NO permite aplicar porcentajes o fórmulas masivas.
       * Importación desde Excel
       * Aplicación masiva de categorías
   * **IDs únicos:** IngresoManual usa prefijo "IM-" y GastoManual usa prefijo "GM-" para garantizar no colisión entre tablas y facilitar referencias cruzadas.
   * *Por qué es importante:* Registro completo de todos los flujos de caja con eficiencia operativa.
   * *Cómo funciona a alto nivel:* Formularios optimizados para entrada rápida, con capacidad de procesamiento por lotes.

* **Gestión de Garantías (Integrada con Contratos, Ingresos/Gastos y Conciliación):**
   * *Qué hace:* Gestiona el ciclo de vida completo de las garantías de arriendo con trazabilidad completa. Las garantías NO generan intereses ni reajustes, se devuelve exactamente el mismo monto recibido. No hay límite mínimo para operar con garantía parcial.
       * **Configuración en Contrato:** Define el `garantia_monto_pactado`. Los demás campos se calculan automáticamente basados en transacciones.
       * **Historial de Garantías:** Nueva tabla `HistorialGarantia` que registra cada transacción con campos unificados y IDs únicos sin colisión:
           * `tipo_movimiento`: 'INGRESO' o 'EGRESO'
           * `movimiento_id`: FK única al IngresoManual o GastoManual correspondiente
           * Proporciona trazabilidad completa
       * **Recepción de Garantía Parcial:** 
           * El contrato puede operar normalmente con garantía parcial de cualquier monto
           * Categorizar ingreso como "Recepción de Garantía del Contrato"
           * Seleccionar Propiedad y Contrato específico
           * Si el monto difiere del pactado, sistema solicita confirmación
           * Actualización automática acumulativa en Contrato
           * Registro en HistorialGarantia
           * Alerta visual permanente de garantía incompleta en dashboard y ficha de contrato
       * **Garantía Excedente**: Si se recibe garantía mayor a la pactada, aplica exactamente el mismo proceso de devolución total que los pagos excedentes (crear IngresoDesconocido + GastoManual por devolución total).
       * **Devolución de Garantía:**
           * Categorizar gasto como "Devolución de Garantía del Contrato"
           * Justificación obligatoria para devoluciones parciales
           * Se devuelve exactamente el monto recibido, sin ajustes
           * Actualización automática de estado
       * **Estados de Garantía:** Calculados automáticamente según montos acumulados
   * *Por qué es importante:* Control total sobre garantías con respaldo completo para auditorías.
   * *Cómo funciona a alto nivel:* Sistema integrado que vincula transacciones categorizadas con contratos, manteniendo campos calculados y trazabilidad.

* **Gestión de Contratos - Funcionalidades Especiales:**
   * **Terminación Anticipada:** Permite finalizar contratos antes de tiempo con actualización automática de pagos futuros.
   * **Plantillas de Contrato:** Los contratos copian valores de plantillas al momento de creación. Cambios en plantillas no afectan contratos existentes.
   * **Versionado:** Cada modificación importante genera una nueva versión con historial completo.
   * **Gestión Documental:** Almacenamiento obligatorio del PDF del contrato firmado con firma electrónica.
   * **Contratos Futuros:** Solo puede existir UN contrato futuro por propiedad. Cuando hay conflicto con renovación automática, el Administrador decide manualmente.
   * **Cambio de Arrendatario:** Permite crear nuevo contrato o transferir el existente al nuevo titular según decisión del Administrador. Si existe contrato futuro, el cambio de titular se permite y el contrato futuro permanece intacto.
   * **Concurrencia de Modificaciones:** Primera operación bloquea el contrato hasta terminar (bloqueo pesimista). Renovación automática, exportaciones y otras modificaciones esperan hasta liberación del lock.

# Technical Architecture

* **System Components:**
   * Backend: Django + Django REST Framework
   * Frontend: React + Vite
   * Database: PostgreSQL con índices optimizados
   * Cache: Redis (aplicación + message broker) con fallback a modo degradado
   * Task Queue: Celery (solo cuando >500 propiedades)
   * Web Scraping: Servicio separado con Playwright/Selenium
   * File Storage: Amazon S3 o Google Cloud Storage
   * API Gateway: Nginx reverse proxy
* **Data Models Principales:**
   * `Socio`, `Empresa`, `Propiedad`, `Arrendatario` (con dirección obligatoria)
   * `CuentaBancaria`, `CredencialBancaria` (con recuperación por preguntas, 2 de 3)
   * `Contrato` (con gestión documental, monto mínimo $1,000), `ContratoVersion`, `PeriodoContractual`
   * `ContratoConTramos`, `TramoArriendo` (para contratos con valores diferenciados)
   * `AjusteContrato` (unificado, por mes completo, solo admin, validación monto mínimo resultado >= $1,000)

   * `HistorialGarantia` (tipo_movimiento + movimiento_id único)

# Risks and Mitigations

* **API Banco no disponible temporalmente:** Sistema automático de fallback a scraping, notificación de estado al admin, conciliación manual como plan C
* **Cambios en sitios scrapeados:** Arquitectura modular, detección de cambios, mantenimiento ágil
* **Seguridad credenciales:** Encriptación AES-256-GCM, acceso solo admin, auditoría total, recuperación por preguntas (2 de 3)
* **Performance con crecimiento:** Caché con fallback desde día 1, alertas preventivas, migración transparente programada
* **Adopción WhatsApp:** Templates pre-aprobados, sin opt-out, notificaciones persistentes siempre 10:00 AM, alerta tras 10 fallos
* **Complejidad operativa:** UI intuitiva en español, wizards, documentación, mensajes claros
* **Redis caído:** Modo degradado automático con funcionalidad básica operativa (contratos actuales + caché HTML)
